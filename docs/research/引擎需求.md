# 三国杀游戏引擎需求分析

本文档基于三国杀标准版规则，分析游戏引擎需要支持的所有机制。

---

## 1. 触发时机系统

### 1.1 回合相关触发时机

#### 1.1.1 回合开始 (onTurnStart)

**场景描述**: 一名角色的回合即将开始时，在准备阶段之前触发。

**具体要求**:
- 在准备阶段开始前触发
- 按座位顺序依次触发被动技能
- 支持"准备阶段开始时"的精确时机

**对引擎的影响**:
- 需要在 `nextPhase` 函数中增加回合开始钩子
- 需要支持技能的优先级排序
- 影响角色: 诸葛亮(观星)、甄姬(洛神)、张辽(突袭)、许褚(裸衣)

**当前实现状态**: 部分实现，`onTurnStart` 触发器已定义但未完整实现

#### 1.1.2 回合结束 (onTurnEnd)

**场景描述**: 一名角色的回合即将结束时，在结束阶段之后触发。

**具体要求**:
- 在结束阶段完成后触发
- 支持跳过弃牌阶段的判定（如吕蒙的克己）
- 支持结束阶段摸牌（如貂蝉的闭月）

**对引擎的影响**:
- 需要在 `nextPhase` 函数中增加回合结束钩子
- 需要支持阶段跳过机制
- 影响角色: 吕蒙(克己)、貂蝉(闭月)

**当前实现状态**: `onTurnEnd` 触发器已定义，克己技能部分实现

#### 1.1.3 阶段开始 (onPhaseStart)

**场景描述**: 每个阶段开始时触发。

**具体要求**:
- 支持6个阶段: 准备、判定、摸牌、出牌、弃牌、结束
- 阶段开始时检查延时锦囊（判定阶段）
- 阶段开始时触发相关技能

**对引擎的影响**:
- 需要在 `nextPhase` 函数中增加阶段开始钩子
- 需要支持延时锦囊的判定流程
- 影响角色: 诸葛亮(观星-准备阶段)、大乔(乐不思蜀-判定阶段)

**当前实现状态**: `phaseChange` 日志已记录，但无阶段开始钩子

#### 1.1.4 阶段结束 (onPhaseEnd)

**场景描述**: 每个阶段结束时触发。

**具体要求**:
- 支持阶段结束时的清理操作
- 支持阶段结束时的技能触发
- 支持跳过下一阶段的判定

**对引擎的影响**:
- 需要在 `nextPhase` 函数中增加阶段结束钩子
- 影响角色: 吕蒙(克己-弃牌阶段结束)

**当前实现状态**: 未实现

### 1.2 卡牌相关触发时机

#### 1.2.1 使用卡牌 (onCardPlayed)

**场景描述**: 一名角色使用卡牌时触发。

**具体要求**:
- 在卡牌从手牌移除前触发
- 支持卡牌类型判定（基本牌、锦囊牌、装备牌）
- 支持卡牌使用合法性校验
- 支持卡牌效果修改

**对引擎的影响**:
- 需要在 `playCard` 函数中增加卡牌使用钩子
- 需要支持卡牌效果修改器
- 影响角色: 黄月英(集智)、司马懿(鬼才)

**当前实现状态**: `onCardPlayed` 触发器已定义，集智技能部分实现

#### 1.2.2 成为目标 (onTargeted)

**场景描述**: 一名角色成为卡牌或技能的目标时触发。

**具体要求**:
- 在卡牌效果结算前触发
- 支持目标合法性校验（如空城）
- 支持目标转移（如流离）
- 支持目标免疫（如谦逊）

**对引擎的影响**:
- 需要在卡牌效果执行前增加目标校验钩子
- 需要支持目标转移机制
- 影响角色: 诸葛亮(空城)、大乔(流离)、陆逊(谦逊)

**当前实现状态**: `onTargeted` 触发器已定义，但未完整实现

#### 1.2.3 卡牌生效 (onCardEffect)

**场景描述**: 卡牌效果即将生效时触发。

**具体要求**:
- 在卡牌效果实际执行前触发
- 支持效果抵消（如无懈可击）
- 支持效果修改（如伤害+1）

**对引擎的影响**:
- 需要在卡牌效果执行前增加效果校验钩子
- 需要支持无懈可击的响应窗口
- 影响角色: 所有使用无懈可击的角色

**当前实现状态**: 未实现

#### 1.2.4 卡牌被抵消 (onCardCancelled)

**场景描述**: 卡牌效果被抵消时触发。

**具体要求**:
- 记录卡牌被抵消的事件
- 触发相关技能（如青龙偃月刀）
- 将卡牌置入弃牌堆

**对引擎的影响**:
- 需要增加卡牌抵消事件
- 需要支持武器技能的触发
- 影响角色: 使用青龙偃月刀的角色

**当前实现状态**: 未实现

### 1.3 伤害相关触发时机

#### 1.3.1 造成伤害 (onDamageDealt)

**场景描述**: 一名角色即将造成伤害时触发。

**具体要求**:
- 在伤害实际生效前触发
- 支持伤害来源判定
- 支持伤害类型判定（普通伤害、属性伤害）
- 支持伤害加成（如裸衣）

**对引擎的影响**:
- 需要在伤害计算前增加伤害来源钩子
- 需要支持伤害类型系统
- 影响角色: 许褚(裸衣)、马超(铁骑)

**当前实现状态**: `onDamageDealt` 触发器已定义，但未完整实现

#### 1.3.2 受到伤害 (onDamageReceived)

**场景描述**: 一名角色受到伤害后触发。

**具体要求**:
- 在伤害实际生效后触发
- 支持伤害来源追踪
- 支持伤害牌追踪
- 支持受伤后的技能触发

**对引擎的影响**:
- 需要在伤害结算后增加受伤钩子
- 需要支持伤害上下文传递
- 影响角色: 曹操(奸雄)、司马懿(反馈)、夏侯惇(刚烈)、郭嘉(遗计)

**当前实现状态**: `onDamageReceived` 触发器已定义，奸雄、反馈、遗计已实现

#### 1.3.3 伤害结算后 (onDamageResolved)

**场景描述**: 伤害完全结算后触发。

**具体要求**:
- 在所有伤害相关技能触发后触发
- 用于清理伤害上下文
- 用于触发延迟效果

**对引擎的影响**:
- 需要增加伤害结算完成事件
- 用于支持复杂的伤害交互

**当前实现状态**: 未实现

### 1.4 状态相关触发时机

#### 1.4.1 濒死 (onDying)

**场景描述**: 一名角色体力值降至0或更低时触发。

**具体要求**:
- 在角色死亡前触发
- 支持自救（使用桃）
- 支持他救（其他角色使用桃）
- 支持多次濒死结算（如连环伤害）

**对引擎的影响**:
- 需要在 `applyDying` 函数前增加濒死钩子
- 需要支持桃的响应窗口
- 影响角色: 所有可以使用桃的角色

**当前实现状态**: `dying.ts` 已实现基本濒死流程

#### 1.4.2 死亡 (onDeath)

**场景描述**: 一名角色死亡时触发。

**具体要求**:
- 在角色死亡后触发
- 支持死亡时技能触发（如死亡时弃牌）
- 支持胜利条件判定
- 支持身份揭示

**对引擎的影响**:
- 需要在 `applyDying` 函数后增加死亡钩子
- 需要支持胜利条件检查
- 影响角色: 所有死亡时触发技能的角色

**当前实现状态**: `playerDeath` 函数已实现基本死亡流程

#### 1.4.3 体力变化 (onHealthChange)

**场景描述**: 一名角色体力值发生变化时触发。

**具体要求**:
- 在体力值实际变化前触发
- 支持体力增加和减少
- 支持体力上限变化
- 支持体力变化后的技能触发

**对引擎的影响**:
- 需要增加体力变化事件
- 需要支持体力变化的来源追踪
- 影响角色: 孙权(救援)

**当前实现状态**: 未实现

### 1.5 判定相关触发时机

#### 1.5.1 判定牌翻开 (onJudgeReveal)

**场景描述**: 判定牌从牌堆顶翻开时触发。

**具体要求**:

- 支持判定牌替换（如鬼才）
- 支持判定牌查看（如天妒）

**对引擎的影响**:

- 需要支持判定牌替换机制
- 影响角色: 司马懿(鬼才)、郭嘉(天妒)

**当前实现状态**: `onJudge` 触发器已定义，但未完整实现

#### 1.5.2 判定结果生效 (onJudgeEffect)

**场景描述**: 判定结果即将生效时触发。

**具体要求**:
- 在判定结果实际生效前触发
- 支持判定结果修改
- 支持判定结果生效后的技能触发

**对引擎的影响**:
- 需要增加判定结果生效事件
- 需要支持判定结果修改器
- 影响角色: 郭嘉(天妒)

**当前实现状态**: 未实现

## 2. 响应窗口系统

### 2.1 杀 → 闪 响应

**场景描述**: 当一名角色使用【杀】指定目标后，目标角色可以使用【闪】来抵消此【杀】。

**具体要求**:
- 响应窗口: 目标角色
- 响应时间: 限时响应（建议10秒）
- 响应选项: 使用【闪】或不响应
- 特殊情况:
  - 八卦阵: 可以进行判定，若结果为红色，视为使用【闪】
  - 无双: 需要两张【闪】才能抵消
  - 龙胆: 可以将【杀】当【闪】使用

**对引擎的影响**:
- 需要实现 `respond_kill` 输入请求
- 需要支持闪的响应窗口
- 需要支持八卦阵的判定流程
- 需要支持无双的多次响应

**当前实现状态**: 基本实现，`respondToKill` 函数已支持闪响应

### 2.2 锦囊 → 无懈可击 响应

**场景描述**: 当一张锦囊牌对目标角色生效前，任何角色可以使用【无懈可击】来抵消此锦囊牌对该角色的效果。

**具体要求**:
- 响应窗口: 所有角色
- 响应时间: 限时响应（建议10秒）
- 响应选项: 使用【无懈可击】或不响应
- 特殊情况:
  - 无懈可击可以抵消无懈可击
  - 响应顺序: 按座位顺序响应

**对引擎的影响**:
- 需要实现 `respond_trick` 输入请求
- 需要支持无懈可击的响应窗口
- 需要支持无懈可击的连锁响应
- 需要支持响应顺序管理

**当前实现状态**: 未实现

### 2.3 濒死 → 桃 响应

**场景描述**: 当一名角色濒死时，任何角色可以使用【桃】来救助该角色。

**具体要求**:
- 响应窗口: 所有角色（按座位顺序）
- 响应时间: 限时响应（建议15秒）
- 响应选项: 使用【桃】或不响应
- 特殊情况:
  - 濒死角色可以自救（使用自己的桃）
  - 救援顺序: 按座位顺序，从濒死角色开始
  - 需要足够的桃才能救活（体力值需要回到1）

**对引擎的影响**:
- 需要实现 `respond_dying` 输入请求
- 需要支持桃的响应窗口
- 需要支持自救和他救
- 需要支持救援顺序管理

**当前实现状态**: 基本实现，`respondToDying` 函数已支持桃响应

### 2.4 万箭齐发/南蛮入侵 → 杀/闪 响应

**场景描述**: 当一名角色使用【万箭齐发】或【南蛮入侵】时，每名目标角色需要响应。

**具体要求**:
- 响应窗口: 每名目标角色（按座位顺序）
- 响应时间: 限时响应（建议10秒）
- 响应选项:
  - 万箭齐发: 使用【闪】或受到1点伤害
  - 南蛮入侵: 使用【杀】或受到1点伤害
- 特殊情况:
  - 龙胆: 可以将【杀】当【闪】使用
  - 倾国: 可以将黑色手牌当【闪】使用
  - 武圣: 可以将红色手牌当【杀】使用

**对引擎的影响**:
- 需要实现 `respond_aoe` 输入请求
- 需要支持群体响应窗口
- 需要支持卡牌转换技能
- 需要支持响应顺序管理

**当前实现状态**: 未实现，当前简化为直接造成伤害

### 2.5 决斗 → 杀 响应

**场景描述**: 当一名角色使用【决斗】时，目标角色需要轮流出【杀】。

**具体要求**:
- 响应窗口: 目标角色和使用者轮流响应
- 响应时间: 限时响应（建议10秒）
- 响应选项: 使用【杀】或不响应（受到1点伤害）
- 特殊情况:
  - 无双: 每次需要出两张【杀】
  - 响应顺序: 由目标角色开始，轮流响应

**对引擎的影响**:
- 需要实现 `respond_duel` 输入请求
- 需要支持轮流响应机制
- 需要支持无双的多次响应
- 需要支持响应顺序管理

**当前实现状态**: 未实现，当前简化为直接造成伤害

### 2.6 延时锦囊判定响应

**场景描述**: 当一名角色的判定阶段需要判定延时锦囊时。

**具体要求**:
- 响应窗口: 当前角色
- 响应时间: 无限制（等待判定结果）
- 响应选项: 使用技能修改判定结果（如鬼才）
- 特殊情况:
  - 鬼才: 可以用手牌替换判定牌
  - 天妒: 可以获得判定牌

**对引擎的影响**:
- 需要实现判定流程
- 需要支持判定牌替换
- 需要支持判定结果生效

**当前实现状态**: 未实现

## 3. 合法性校验

### 3.1 距离校验

#### 3.1.1 杀的距离校验

**场景描述**: 使用【杀】时，目标必须在攻击范围内。

**具体要求**:
- 攻击范围 = 武器攻击范围（默认1）
- 距离计算: 两个玩家之间存活玩家数的最小值
- 距离修正:
  - -1马: 减少你到别人的距离
  - +1马: 增加别人到你的距离
- 特殊情况:
  - 奇才: 使用锦囊牌无距离限制

**对引擎的影响**:
- `getDistance` 函数已实现基本距离计算
- `getAttackRange` 函数已实现攻击范围计算
- 需要支持距离修正的叠加

**当前实现状态**: 基本实现

#### 3.1.2 顺手牵羊的距离校验

**场景描述**: 使用【顺手牵羊】时，目标必须在距离1以内。

**具体要求**:
- 距离计算: 同杀的距离计算
- 距离限制: 距离 <= 1

**对引擎的影响**:
- `getValidTargetsForCard` 函数已实现距离校验
- 需要支持距离限制的配置化

**当前实现状态**: 基本实现

#### 3.1.3 乐不思蜀/兵粮寸断的距离校验

**场景描述**: 使用【乐不思蜀】或【兵粮寸断】时，目标必须在距离1以内。

**具体要求**:
- 距离计算: 同杀的距离计算
- 距离限制: 距离 <= 1

**对引擎的影响**:
- `getValidTargetsForCard` 函数需要增加延时锦囊的距离校验
- 当前实现未校验距离

**当前实现状态**: 未实现

### 3.2 目标校验

#### 3.2.1 不能对自己使用

**场景描述**: 大部分卡牌不能对自己使用。

**具体要求**:
- 不能对自己使用的卡牌: 杀、过河拆桥、顺手牵羊、决斗、乐不思蜀、兵粮寸断
- 可以对自己使用的卡牌: 桃、无中生有、桃园结义、装备牌

**对引擎的影响**:
- `getValidTargetsForCard` 函数已排除自己
- 需要支持卡牌的目标限制配置

**当前实现状态**: 基本实现

#### 3.2.2 不能对已死亡玩家使用

**场景描述**: 不能对已死亡的玩家使用卡牌。

**具体要求**:
- 所有卡牌都不能对已死亡玩家使用
- 死亡玩家不能使用卡牌

**对引擎的影响**:
- `getValidTargetsForCard` 函数已过滤死亡玩家
- 需要支持死亡状态检查

**当前实现状态**: 基本实现

#### 3.2.3 不能成为特定卡牌的目标

**场景描述**: 某些角色不能成为特定卡牌的目标。

**具体要求**:
- 空城: 没有手牌时，不能成为【杀】或【决斗】的目标
- 谦逊: 不能成为【过河拆桥】和【顺手牵羊】的目标

**对引擎的影响**:
- 需要在目标校验时检查目标角色的技能
- 需要支持目标免疫机制

**当前实现状态**: 未实现

#### 3.2.4 需要特定条件才能成为目标

**场景描述**: 某些卡牌需要目标满足特定条件。

**具体要求**:
- 过河拆桥/顺手牵羊: 目标需要有牌（手牌或装备）
- 桃: 只能对受伤角色使用（自己或濒死角色）

**对引擎的影响**:
- `getValidTargetsForCard` 函数已检查目标是否有牌
- 需要支持条件化目标校验

**当前实现状态**: 基本实现

### 3.3 卡牌使用限制

#### 3.3.1 每回合杀的数量限制

**场景描述**: 每回合只能使用一张【杀】。

**具体要求**:
- 默认限制: 每回合1张
- 特殊情况:
  - 诸葛连弩: 取消杀的数量限制
  - 咆哮: 取消杀的数量限制

**对引擎的影响**:
- 需要跟踪本回合使用的杀的数量
- 需要支持杀的数量限制的修改

## 6. 优先级与实现建议

### 6.1 高优先级（核心游戏流程）

1. **触发时机系统**: 实现完整的触发时机系统，支持所有触发时机
2. **响应窗口系统**: 实现杀→闪、锦囊→无懈可击、濒死→桃的响应窗口
3. **合法性校验**: 实现距离、目标、卡牌使用限制的校验
4. **被动技能触发钩子**: 实现伤害、卡牌使用、回合相关的钩子

### 6.2 中优先级（扩展功能）

1. **卡牌效果修改钩子**: 实现伤害、卡牌效果的修改
2. **距离计算修改钩子**: 实现距离、攻击范围的修改
3. **前端交互**: 实现目标选择、卡牌选择、响应提示

### 6.3 低优先级（完善功能）

1. **判定系统**: 实现完整的判定流程
2. **复杂技能**: 实现复杂的技能交互
3. **优化**: 性能优化、用户体验优化

## 7. 架构建议

### 7.1 事件驱动架构

建议采用事件驱动架构，将游戏逻辑分解为事件和事件处理器：

```typescript
interface GameEvent {
  type: string;
  data: unknown;
  timestamp: number;
}

interface EventHandler {
  handle(event: GameEvent, state: GameState): GameState;
}

class EventBus {
  private handlers: Map<string, EventHandler[]> = new Map();
  
  on(type: string, handler: EventHandler): void;
  emit(event: GameEvent, state: GameState): GameState;
}
```

### 7.2 钩子系统

建议实现统一的钩子系统，支持优先级和取消：

```typescript
interface Hook<T> {
  priority: number;
  handler: (context: T) => T | null; // null表示取消
}

class HookSystem {
  private hooks: Map<string, Hook<unknown>[]> = new Map();
  
  register<T>(name: string, hook: Hook<T>): void;
  execute<T>(name: string, context: T): T;
}
```

### 7.3 状态机

建议使用状态机管理游戏状态转换：

```typescript
interface GameStateMachine {
  currentState: GameState;
  transitions: Map<string, Transition>;
  
  transition(event: string): GameState;
  canTransition(event: string): boolean;
}
```

### 7.4 响应队列

建议实现响应队列管理玩家响应：

```typescript
interface ResponseQueue {
  queue: Array<{
    player: string;
    type: string;
    timeout: number;
  }>;
  
  add(player: string, type: string, timeout: number): void;
  remove(player: string): void;
  getNext(): { player: string; type: string } | null;
}
```

## 8. 总结

三国杀游戏引擎需要支持以下核心机制：

1. **触发时机系统**: 支持回合、卡牌、伤害、状态、判定相关的所有触发时机
2. **响应窗口系统**: 支持杀→闪、锦囊→无懈可击、濒死→桃等响应场景
3. **合法性校验**: 支持距离、目标、卡牌使用限制的校验
4. **前端交互**: 支持目标选择、卡牌选择、响应提示、信息展示
5. **引擎钩子**: 支持被动技能触发、卡牌效果修改、距离计算修改、伤害计算修改
