# 引擎缺失能力

本文档记录当前游戏引擎尚不支持的能力,以及每个能力对应的被跳过技能。

> 创建时间:2026-06-14
> 核心结论:**引擎执行模型能表达全部三国杀技能**,没有任何需要打破"同时只一个 pending"不变量的情况。下述缺口全部是数据层(atom/字段)或技能层(execute 逻辑)的加法。
>
> ---
> **【更新 2026-07-08】** 全文逐项代码核实完成。结论:**17 项中 16 项已实现,仅第 12 项(于吉·蛊惑 / 多玩家质疑)尚未实现**。下述各节均保留 2026-06-14 原始判断,并在节首追加核实结果标注。详见文末「核实总览」。

## 执行模型结论

经过对无懈可击、濒死求桃、于吉蛊惑(扣牌+揭示)等"最难"技能的推演验证:

- **多人响应**(求桃/无懈/质疑)都是**依次串行**,不是并发——单槽 pending + `for` 循环 + 状态观察足够表达
- **抢占式响应**(无懈可击)是数据层约定——`请求回应` 声明合法回应者集合(不限单一 target),仍保持同时只一个 pending
- **隐藏信息**(扣牌/展示)用 `toViewEvents` 的 `ownerViews`/`othersView` 分级——不返回给不该看的客户端即可
- **HookResult**(pass/modify/cancel)覆盖了事件取消/变更(仁王盾、藤甲、寒冰剑)的全部需求

因此下述各项**均为数据层缺口**(补 atom、补字段、写技能 execute),不涉及执行模型改动。

**【更新 2026-07-08】** 此结论已被实现充分验证:濒死求桃、无懈可击、创牌、化身、乱武等最难项均在不改动执行模型的前提下落地,执行模型至今未变。

## 1. 濒死/求桃流程(核心规则)— ✅ 已实现

**【更新 2026-07-08】已实现。**
- atom:`陷入濒死`(`src/engine/atoms/陷入濒死.ts`,纯事件标记)+ `击杀` atom。
- 濒死循环:`系统规则.ts` 的 `runDyingFlow`——`造成伤害`/`失去体力` after-hook 在 `target.health <= 0` 时触发,先 `applyAtom(陷入濒死)`,再按座次 `for` 循环逐个 `applyAtom(请求回应-{requestType:'桃/求桃'})`,有人救则 `回复体力` 脱离,无人救才 `击杀`。
- 测试:`tests/integration/濒死求桃.test.ts`。
- 不屈(周泰)在 `runDyingFlow` 内通过 `localVars['不屈/存活']` 标记劫持,跳过求桃+击杀(见第 13 项)。

**影响**:几乎每局(桃、酒救援、郭嘉遗计死亡时机、贾诩完杀、周泰不屈)

**现状**:`造成伤害.apply` 扣到 0 直接 `alive=false`,无濒死窗口。

**需要**:
- 新 atom `进入濒死`(target 扣到 0 时触发)
- 濒死是逐个询问的 pending 循环:`for (player of 轮转) await applyAtom(请求回应-桃, target: player)`,有人救则脱离
- `造成伤害` 的 after 钩子(或改为 before 钩子 cancel)扣到 0 时进入濒死循环,无人救才 `击杀`

## 3. 无懈可击 / 锦囊生效前拦截 — ✅ 已实现

**【更新 2026-07-08】已实现。**
- 技能模块:`src/engine/skills/无懈可击.ts`;复用辅助函数 `询问无懈可击(state, target)`(被 `决斗.ts` 等锦囊 execute 调用)。
- 广播 pending + 合法回应者集合:`请求回应` 放宽为允许声明 `respondable`(广播型用 `target === TARGET_BROADCAST`,任意玩家 respond 命中同一 slot)。
- 套娃轮询:锦囊 execute 内 `while` 轮询 `询问无懈可击`,奇偶判定生效/抵消。
- 测试:`tests/integration/无懈可击.test.ts`。

**影响**:无懈可击(及其套娃)、所有锦囊的"询问无懈"环节

**现状**:`无懈可击.ts` 注释自承"整体不工作"——锦囊 execute 没有"询问无懈"循环。

**需要**:
- `dispatch` 回应者校验放宽:`请求回应` 加 `respondable?: number[]` 字段(默认 `[target]`),无懈类设为全体活着的玩家;`dispatch:237` 从 `!== target` 改为 `!respondable.includes(ownerId)`
- 锦囊 execute 写无懈轮询循环:`while (true) { applyAtom(请求回应-无懈); if 无人 break; count++; }`,奇偶判定生效
- 多目标锦囊:无懈的 prompt/params 带目标信息,respond 时选(数据层)

## 4. 阶段跳过/替换 — ✅ 已实现(HookResult cancel)

**【更新 2026-07-08】已实现。** 阶段跳过统一采用「`applyAtom(阶段结束, 当前阶段)` 推进 + before-hook `return { kind: 'cancel' }` 取消本次 `阶段开始`」手法,`回合管理` after-hook(自动摸2张)不再执行:
- `神速.ts`(夏侯渊):选项1 判定/摸牌阶段 cancel、选项2 出牌阶段 cancel + 虚拟杀。
- `巧变.ts`(张郃):判定/摸牌/出牌/弃牌 before-hook,跳过 + 各自附加效果。
- `双雄.ts`(颜良文丑):摸牌阶段 cancel + 展示牌堆顶2张选一 + 转化决斗。
- `流离.ts`(大乔):实为「成为目标」after-hook 修改杀帧 `resolvedTargets`(目标转移),非阶段跳过,亦已实现。
- 测试:`tests/integration/流离.test.ts`、`tests/integration/乐不思蜀判定跳过.test.ts`。

**影响**:夏侯渊神速、张郃巧变、颜良文丑双雄、大乔流离

**现状**:阶段开始/结束 atom 有,回合管理硬编码阶段序列。

**需要**:`阶段开始` 的 before hook 返回 `cancel` → 跳过该阶段。`回合管理.ts` 从"硬编码 apply 序列"改为"读 state.phase 驱动,尊重 cancel"。HookResult 机制已就绪。
