> 技能开发团队以此为基准对齐设计，先规范后迁移。

## 一、Atom 清单 ⚠️ 清单过时（25→37 个，缺 P0 新增的 giveCard/takeCard/reshuffle/useCard 3原子/compareRank 等），以 engine/types.ts 为准。但每个 atom 的"前端指令/动画映射"仍有参考价值。

### 1.1 什么是 Atom

Atom 是**唯一修改 GameState 的通道**，也是**连接引擎和后端的消息单元**。

每个 Atom 包含两面：

```
Atom = { apply(state) → state, toEvents(state) → [serverEvent, playerViews] }
```

- `apply`：纯状态转移函数，不产生副作用
- `toEvents`：为每个玩家生成视角差异化的事件（公开/隐藏信息）

### 1.2 完整 Atom 清单

| # | Atom | 后端效果 | 前端指令 | 是否需要辅助操作 |
|---|---|---|---|---|
| 1 | `damage` | target.health -= amount | 扣血动画、掉血音效 | 否 |
| 2 | `heal` | target.health += amount | 回血动画、回血音效 | 否 |
| 3 | `draw` | deck → player.hand | 摸牌动画、手牌+1（自己看到牌面，他人看到数量） | 否 |
| 4 | `discard` | player.hand → discardPile | 弃牌动画、手牌-1 | 是（选择弃哪些牌） |
| 5 | `discardRandom` | player.hand → discardPile（随机选） | 弃牌动画 | 否 |
| 6 | `moveCard` | card 从源区域移到目标区域 | 移牌动画 | 否 |
| 7 | `equip` | card 从手牌→装备区 | 装备动画、属性变化 | 否 |
| 8 | `unequip` | card 从装备区→弃牌堆 | 卸装动画 | 否 |
| 9 | `setVar` | player.vars[key] = value | 状态变化（通常无前端展示） | 否 |
| 10 | `incrementVar` | player.vars[key] += delta | 同上 | 否 |
| 11 | `clearVarPattern` | 按 pattern 清除 vars | 同上 | 否 |

| 13 | `pushPending` | 暂停引擎，等待用户输入 | 弹窗/选择界面 | 是（定义 pending 类型） |
| 14 | `popPending` | 恢复引擎执行 | 关闭弹窗 | 否 |
| 15 | `setPhase` | state.phase = phase | 阶段切换 UI | 否 |
| 16 | `nextPlayer` | 轮到下一位玩家 | 回合切换动画 | 否 |

### 1.3 Atom 测试规范

每个 Atom 需要两种测试：

**后端测试：apply 状态转移**
```
describe('damage atom', () => {
  it('target health 减少 amount', () => {
    const state = createBaseState({ P1: { health: 3 } })
    const next = applyAtom(state, { type: 'damage', target: 'P1', amount: 2 })
    expect(next.players.P1.health).toBe(1)
  })
})
```

**前端测试：toEvents 视角分发**
```
describe('draw atom toEvents', () => {
  it('摸牌玩家看到牌面，其他人看到数量', () => {
    const [server, special, defaultEvt] = drawDef.toEvents(state, atom)
    expect(special.get('P1')?.cards).toHaveLength(2)  // P1 看到具体牌
    expect(defaultEvt?.cards).toBeUndefined()           // 他人只看数量
  })
})
```

## 二、Handler 纯化契约

### 2.1 契约定义

**每条规则都是强制的。违反任意一条即为架构违规。**

```
handler(ctx, state): SkillPhase[]
```

| # | 规则 | 禁止 | 允许 |
|---|---|---|---|
| R1 | Handler 必须是纯函数 | 调用 `emitEvent`、`applyAtom`、`applyDamage` | 仅返回 `SkillPhase[]` |
| R2 | Handler 可以读取 state，不可写入 state | 修改 state、调用 mutable 方法 | 通过 `ctx` 和 `state` 参数只读访问 |
| R3 | 状态变更必须通过 atoms | 直接修改 `state.players`、`state.zones` | 返回 `{ type: 'atoms', ops: [...] }` |
| R4 | 复杂流程必须用 phase 编排 | 在 handler 内写 for/while 遍历 | 使用 `condition`/`loop` phase |
| R5 | 用户交互必须用 prompt phase | 直接调用 UI 或设置 pending | 返回 `{ type: 'prompt', ... }` 由引擎调度 |

### 2.2 设计动机

Handler 纯化彻底解耦"决定要做什么"和"执行这些操作"：

```
技能触发 → handler(state, ctx) → SkillPhase[]  // 决策
              ↓
          executePlan(phases)  →  状态变更     // 执行
```

好处：
1. **可测试**：`handler(state, ctx)` 的输入和输出都是纯数据，无需 mock 引擎
2. **可追踪**：所有的 SkillPhase 构成了技能的"执行计划"，可以序列化、记录、回放
3. **可中断**：prompt phase 暂停后，resume 只需从上次位置继续执行剩余 phases
4. **可组合**：condition/loop 可以嵌套组合任意复杂逻辑

### 2.3 当前违规点及迁移方案

| 技能 | 违规 | 迁移方法 |
|---|---|---|
| 苦肉 | handler 使用 `checkDying` phase（实际上是合法的 phase） | 合法，无需迁移。但 engine 执行 checkDying 时需要发射 damageReceived 事件 |
| 观星 | handler 调用 `buildRearrangeTree()`（递归构建 phases） | 合法，纯函数递归构建 phases 树 |
| 青囊 | handler 在 phase 内使用 `setVar` 设置 usedThisTurn | 合法，通过 atoms 修改状态 |
| 反馈 | handler 返回两个 `atoms` phases（discardRandom → TODO gainCard） | 合法但 TODO 未完成 |

**当前真正的问题不是 handler 本身不纯，而是 engine 在 executePlan 时不发射 GameEvent。**

例如苦肉 handler 返回 `[{ type: 'atoms', ops: [{ type: 'damage', ... }] }]`，executePlan 跑完 damage atom 后应该自动触发 `damageReceived` 事件链，但当前代码不会——atoms 只是修改 state，不触发事件。

### 2.4 修复方案：Atom 执行后事件发射

```
executePlan(state, phases, ctx):
  for each phase:
    执行 phase（可能改变 state）
    if phase.type === 'atoms' && 某 atom 需要事件:
      emitEvent({ type: 'damageReceived', ... })  // 自动触发技能链
```

第 1 步定义映射：
```
const ATOM_TO_EVENTS = {
  damage:  [{ type: 'damageReceived', ... }],    // 触发 damage 相关的技能
  heal:    [{ type: 'healed', ... }],
  discard: [{ type: 'cardDiscarded', ... }],
  ...
}
```

第 2 步在 executePlan 中，处理完 atoms phase 后，根据 `ATOM_TO_EVENTS` 自动发射事件。

> ⚠️ 这需要与当前 handler（如 苦肉 手动用 `checkDying` phase）协调。
> 迁移方案见下文 §2.5。

### 2.5 迁移路线

```
阶段 1：文档对齐（本文档）
阶段 2：引擎改造 — executePlan 在 damage atom 后自动发射 damageReceived
阶段 3：检查 dying 机制 — damageReceived 触发濒死检查的时机
阶段 4：逐技能验证 — 每个 skill 的 handler 只返回 phases，不产生副作用
阶段 5：移除对 emitEvent 的 handler 内直接调用
```

### 3.1 原则

- 不测 handler 返回值（那是 TS 类型检查的事）

## 四、前端视图状态与动画验证 ⚠️ 设计蓝图，尚未实现。实际视图层在 engine/view/ 下（PlayerView/FrontendState/reduceGameState），但 Animation/isValidAction/getAvailableActions/eventsToAnimations 未实现。

### 4.1 数据流

```
后端 GameState ──原子──→ 事件流(events) ──→ reduceFrontend ──→ FrontendState
                                                            ├── views[playerId]: PlayerView
                                                            ├── animationQueue: Animation[]
                                                            └── availableActions: AvailableAction[]
```

- 前端仍以**事件流为唯一数据源**，不直接访问后端 GameState
- `PlayerView` 是前端从事件流还原出的 UI 状态快照，测试中可直接断言
- `Animation` 是事件的副作用描述，测试中可断言动画序列
- `AvailableAction` / `isValidAction` 是操作辅助与校验，纯函数可测试
- 整个 `reduceFrontend(frontend, events) → frontend` 是纯函数，无需 React 环境即可测试

### 4.2 类型定义

```typescript
// 每个玩家看到的 UI 状态
interface PlayerView {
  self: {
    hand: CardInfo[];
    equipment: { weapon: CardInfo | null; armor: CardInfo | null; mount: CardInfo | null };
    health: number;
    maxHealth: number;
    pendingTricks: PendingTrickInfo[];
    tags: string[];
    vars: Record<string, Json>;
  };
  others: Record<string, {
    handCount: number;
    equipment: { weapon: string | null; armor: string | null; mount: string | null };
    health: number;
    maxHealth: number;
    pendingTrickCount: number;
    alive: boolean;
  }>;
  table: { discardPileCount: number; deckCount: number };
  turn: { phase: string; currentPlayer: string };
}

// 动画指令（纯数据，前端渲染层消费）
type Animation =
  | { type: 'cardMove'; cardId: string; from: { zone: string; player?: string }; to: { zone: string; player?: string }; duration: number }
  | { type: 'cardFlip'; cardId: string }
  | { type: 'damagePopup'; target: string; amount: number }
  | { type: 'healGlow'; target: string; amount: number }
  | { type: 'drawCards'; player: string; count: number }
  | { type: 'discardCards'; player: string; cardIds: string[] }
  | { type: 'equipItem'; player: string; cardId: string; slot: string }
  | { type: 'unequipItem'; player: string; slot: string }
  | { type: 'death'; player: string }
  | { type: 'skillActivate'; player: string; skillId: string }
  | { type: 'pendingPrompt'; actionType: string }
  | { type: 'trickReveal'; cardId: string; result: 'success' | 'fail' }
  ;

// 前端可用操作
interface AvailableAction {
  type: 'playCard' | 'useSkill' | 'respond' | 'discard';
  sourceId?: string;
  validTargets: string[];
  required: boolean;
  hint?: string;
}

// 前端整体状态（测试中构造和断言）
interface FrontendState {
  views: Record<string, PlayerView>;
  myPlayerId: string;
  animationQueue: Animation[];
  pending: PendingAction | null;
}
```

### 4.3 reduceFrontend：事件 → 前端状态

纯函数，模拟前端从事件流还原 UI 状态的行为。每个原子类型对应一段 reducer 逻辑。

```typescript
function reduceFrontend(frontend: FrontendState, events: PlayerEvent[]): FrontendState
```

reducer 逻辑概要（与 atom 一一对应）：

| Atom | PlayerView 影响 | 信息隐藏 |
|---|---|---|
| `damage` | target 的 health 减少；`damagePopup` 动画入队 | 无 |
| `heal` | target 的 health 增加；`healGlow` 动画入队 | 无 |
| `draw` | 摸牌者 self.hand 推入详情，others[摸牌者].handCount++；`drawCards` 动画入队 | 摸牌者看到牌面，他人只看数量 |
| `discard` | 弃牌者 self.hand 移除，others[弃牌者].handCount--；`discardCards` 动画入队 | 弃牌者知道弃了哪些，他人只看数量 |
| `equip` | self.hand 移除 → self.equipment 替换，旧进 discardPile；`equipItem`/`unequipItem` 动画 | 所有人看到装备 |
| `moveCard` | 根据 from/to zone 更新对应区域 | 取决于区域可见性 |
| `judge` | 判定牌临时展示 → 入 discardPile；`cardFlip` 动画 | 所有人看到结果 |
| `gainCard` | 获得者 self.hand 推入，他人 handCount++；`cardMove` 动画 | 获得者看到牌面，他人只看数量 |
| `kill` | health 置 0，equipment 清空，alive=false；`death` 动画 | 无 |
| `kill` (救回) | health 回复，alive=true | 无 |
| `setPhase`/`nextPlayer` | turn.phase / turn.currentPlayer 更新 | 无 |
| `setVar` | self.vars[key] 更新 | 仅影响自身 |
| `addPendingTrick` | self.pendingTricks 追加；`pendingPrompt` 动画 | 目标看到详情，他人看到有延时 |
| `removePendingTrick` | self.pendingTricks 移除；`trickReveal` 动画 | 同上 |

### 4.4 测试示例

```typescript
// ── reduceFrontend 测试 ──

it('damage 减少目标体力 + 掉血动画入队', () => {
  let fe = createFrontend({ P1: { health: 4 }, P2: { health: 4 } }, 'P1');
  fe = reduceFrontend(fe, [
    makePlayerEvent('damage', { target: 'P2', amount: 1 }),
  ]);
  expect(fe.views['P1'].others['P2'].health).toBe(3);
  expect(fe.animationQueue).toContainEqual(
    { type: 'damagePopup', target: 'P2', amount: 1 }
  );
});

it('draw 摸牌者看到牌面，他人只看数量', () => {
  let fe = createFrontend({ P1: { hand: ['sha-1'] }, P2: { hand: [] } }, 'P2');
  fe = reduceFrontend(fe, [
    makePlayerEvent('draw', { player: 'P1', count: 2, cards: [{ id: 'sha-2', name: '杀' }, { id: 'shan-1', name: '闪' }] }),
  ]);
  // P2 视角看 P1
  expect(fe.views['P2'].others['P1'].handCount).toBe(3);    // 1+2
  // P1 视角看自己（events 中没有 P1 的事件，用实际事件验证）
  const fe2 = reduceFrontend(createFrontend({ P1: { hand: ['sha-1'] }, P2: { hand: [] } }, 'P1'), [
    makePlayerEvent('draw', { player: 'P1', count: 2, cards: [{ id: 'sha-2', name: '杀' }, { id: 'shan-1', name: '闪' }] }),
  ]);
  expect(fe2.views['P1'].self.hand).toHaveLength(3);
  expect(fe2.views['P1'].self.hand[1].name).toBe('杀');
});

it('动画不暴露隐藏信息', () => {
  // 其他人看到的 draw 事件不包含 cards，动画不应包含卡牌详情
  const events = [makePlayerEvent('draw', { player: 'P1', count: 2 })];
  expect(eventsToAnimations('P2', events)).toContainEqual(
    { type: 'drawCards', player: 'P1', count: 2 }
  );
});
```
