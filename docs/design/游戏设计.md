# 三国杀 数字卡牌游戏设计文档

## 概述

构建一个基于 Web 的三国杀多人在线卡牌游戏，采用服务器权威架构，数据驱动的角色技能系统。

**技术栈:** React + TypeScript + Hono + ~~Drizzle~~（未实现，当前使用内存存储） + Vite + pnpm

**平台:** Web 优先，后续可扩展到桌面和移动端

## 状态管理 ⚠️ 过时 — 实际使用 React atoms + Event Sourcing

不使用 Zustand 等状态管理库。采用以下方案:

- **服务器状态:** 通过 plain object 存储，使用 `@for-fun/event-emitter` 通知状态变更
- **本地 UI 状态:** 子组件内部使用 `useState` 管理
- **前端状态不复杂，不需要额外的状态管理依赖**

### 回合阶段

标准三国杀回合流程:

1. **准备阶段** — 触发"回合开始时"技能
2. **判定阶段** — 解决延时锦囊
3. **摸牌阶段** — 摸 2 张牌
4. **出牌阶段** — 使用卡牌、发动技能
5. **弃牌阶段** — 弃牌至手牌上限
6. **结束阶段** — 触发"回合结束时"技能

引擎按阶段顺序执行，需要玩家输入时暂停（如选择出什么牌）。通过 WebSocket 与客户端通信——发送状态更新并等待玩家操作。

### 玩家输入提示（Prompt）

当引擎需要玩家做出选择时，发送 `prompt` 消息并等待 `response`:

```typescript
interface Prompt {
  name: string;            // 唯一标识，如 "选择目标", "选择弃牌"
  description: string;     // 显示给玩家的说明
  type: 'select_player' | 'select_card' | 'select_yes_no' | 'select_option';
  options: any[];          // 可选项列表
  timeout?: number;        // 超时时间（秒），超时自动选择默认值
}
```

**流程:**
1. 引擎遇到需要玩家输入的时机
2. 向该玩家发送 `prompt` 消息
3. 引擎暂停，等待该玩家的 `response`
4. 收到 `response` 后继续执行
5. 如果超时未响应，使用默认值（如自动弃牌、自动选择"否"）

**常见 Prompt 场景:**
- 出牌阶段: 选择要使用的卡牌或技能
- 选择目标: 杀/锦囊需要选择目标玩家
- 响应: 被杀时选择是否出闪，被南蛮入侵时选择是否出杀
- 弃牌: 弃牌阶段选择弃哪些牌
- 技能发动: 被动技能触发时选择是否发动

## 技能与卡牌系统（数据驱动）⚠️ 过时 — 见 ENGINE-DESIGN.md §4 技能（钩子机制 + SkillPhase）

角色技能定义为配置对象，引擎运行时解释执行。不为每个角色编写类。

### 角色配置

```typescript
interface CharacterConfig {
  name: string;              // "曹操", "刘备" — 作为唯一标识
  maxHealth: number;
  gender: '男' | '女';
  faction: '魏' | '蜀' | '吴' | '群';
  abilities: AbilityConfig[];
}

interface AbilityConfig {
  name: string;              // "奸雄", "仁德" — 作为唯一标识
  description: string;
  trigger: TriggerType;
  condition?: Condition;
  effect: Effect;
  oncePerTurn?: boolean;
  passive?: boolean;         // 自动触发 vs 手动发动
}
```

### 触发时机

`onDamageReceived`, `onDamageDealt`, `onTurnStart`, `onTurnEnd`, `onCardPlayed`, `onCardDrawn`, `onKill`, `onDeath`, `manual`

### 效果原语

可组合的基础效果:

- `draw(n)` — 摸 n 张牌
- `dealDamage(target, n)` — 对目标造成 n 点伤害
- `heal(target, n)` — 恢复 n 点体力
- `discard(target, n)` — 弃置 n 张牌
- `gainCard(target, card)` — 获得目标的牌
- `skipPhase(phase)` — 跳过某个阶段
- `conditional(condition, thenEffect, elseEffect)` — 条件分支

### 示例 — 曹操"奸雄"

```typescript
{
  name: '奸雄',
  description: '当你受到伤害后，你可以获得对你造成伤害的牌',
  trigger: 'onDamageReceived',
  effect: {
    type: 'gainCard',
    source: 'damageSourceCard'
  }
}
```

### 示例 — 刘备"仁德"

```typescript
{
  name: '仁德',
  description: '出牌阶段，你可以将任意数量的手牌交给其他角色，每阶段以此法给出两张或更多后，你回复1点体力',
  trigger: 'manual',
  condition: { phase: 'play', hasHandCards: true },
  effect: {
    type: 'sequence',
    steps: [
      { type: 'giveCards', count: 'any', target: 'otherPlayer' },
      {
        type: 'conditional',
        condition: { cardsGivenThisPhase: { gte: 2 } },
        then: { type: 'heal', target: 'self', amount: 1 }
      }
    ]
  }
}
```

## 多人联机与网络

### 房间系统

- 玩家通过 REST API 创建或加入房间
- 房主设置玩家人数（4-8）和游戏选项
- 所有玩家准备后游戏开始
- 房间状态保存在内存中

### WebSocket 协议 ⚠️ 过时 — 见 src/server/protocol.ts

JSON 消息，包含 `type` 字段。实际协议已增加 **EventSeq 序号机制**（CAS baseSeq），
用于客户端去重和断线重传。详见 `src/src/server/protocol.ts` 和 ADR 0009。

```typescript
// ⚠️ 以下为早期设计，实际类型更多
// Server → Client（实际 11 种：含 room_joined/player_joined/reconnect 等）
type ServerMessage =
  | { type: 'initialView'; state: FrontendState; lastSeq: EventSeq }
  | { type: 'events'; fromSeq: EventSeq; events: SequencedEvent[] }
  | { type: 'gameOver'; winner: string }
  | { type: 'error'; message: string }
  | ...

// Client → Server（实际 11 种：含 join_room/create_room/start_game/reconnect 等）
type ClientMessage =
  | { type: 'action'; action: GameAction; baseSeq: EventSeq }
  | { type: 'response'; baseSeq: EventSeq; choice: unknown }
  | { type: 'ready' }
  | ...
```

### 可见性

服务器发送 `FrontendState`（含 `PlayerView` + pending + actions）——每个玩家只能看到自己的手牌和公开信息。详见 `src/src/engine/view/types.ts`。

### 断线重连

- 客户端携带 `lastSeq`，服务端从 `lastSeq+1` 回放后续事件（断点续传）
- 详见 `src/src/server/session.ts` 和 ADR 0009

### 核心组件

- `GameBoard` — 主布局，围绕牌桌摆放玩家
- `PlayerPanel` — 显示角色头像、体力、装备、身份（已揭露时）

- `ActionPanel` — 出牌、发动技能、结束回合按钮
- `PromptModal` — 需要选择时弹出（选择目标、选择卡牌）

## 测试策略

### 单元测试 (Vitest)

- 游戏引擎逻辑——阶段转换、卡牌效果、技能触发
- 技能解释器——每种效果类型产生正确的状态变更
- 牌组管理——洗牌、摸牌、弃牌
- 可见性规则——玩家只能看到自己的手牌
- Atom toEvents——每个 atom 的服务端事件 + 视角裁剪正确性
- View reducer——从 serverLog 推演 ClientGameState 的正确性

### 场景测试 (Vitest)

- `tests/scenarios/{魏,蜀,吴,群,装备,交互}/`——按势力/类型组织的技能场景
- 每个已实现技能至少 1 个端到端场景
- 使用 `tests/scenario-runner.ts` DSL 构建

### 集成测试

- 完整游戏流程——创建房间、加入玩家、完成一局游戏
- WebSocket 消息处理——每个动作发送/接收正确消息
- 多人场景——2-3 名玩家轮流操作
- 服务端 session 生命周期 + 序列号一致性

### 端到端测试 (Playwright)

- 房间创建和加入
- 使用卡牌并查看结果
- 完整游戏会话从开始到结束

### 重放测试

- 序列化 + 反序列化 round-trip
- 从 serverLog 重放还原完整 GameState
- 视角一致性——每个玩家的 PlayerView 与实际状态一致
