# 技能系统测试与架构规范

> 本文档定义三项核心规范：Atom 清单、Handler 纯化契约、场景测试 DSL。
> 技能开发团队以此为基准对齐设计，先规范后迁移。

---

## 一、Atom 清单

### 1.1 什么是 Atom

Atom 是**唯一修改 GameState 的通道**，也是**连接引擎和后端的消息单元**。

每个 Atom 包含两面：

```
Atom = { apply(state) → state, toEvents(state) → [serverEvent, playerViews] }
```

- `apply`：纯状态转移函数，不产生副作用
- `toEvents`：为每个玩家生成视角差异化的事件（公开/隐藏信息）

### 1.2 完整 Atom 清单

| # | Atom | 后端效果 | 前端指令 | 是否需要辅助操作 |
|---|---|---|---|---|
| 1 | `damage` | target.health -= amount | 扣血动画、掉血音效 | 否 |
| 2 | `heal` | target.health += amount | 回血动画、回血音效 | 否 |
| 3 | `draw` | deck → player.hand | 摸牌动画、手牌+1（自己看到牌面，他人看到数量） | 否 |
| 4 | `discard` | player.hand → discardPile | 弃牌动画、手牌-1 | 是（选择弃哪些牌） |
| 5 | `discardRandom` | player.hand → discardPile（随机选） | 弃牌动画 | 否 |
| 6 | `moveCard` | card 从源区域移到目标区域 | 移牌动画 | 否 |
| 7 | `equip` | card 从手牌→装备区 | 装备动画、属性变化 | 否 |
| 8 | `unequip` | card 从装备区→弃牌堆 | 卸装动画 | 否 |
| 9 | `setVar` | player.vars[key] = value | 状态变化（通常无前端展示） | 否 |
| 10 | `incrementVar` | player.vars[key] += delta | 同上 | 否 |
| 11 | `clearVarPattern` | 按 pattern 清除 vars | 同上 | 否 |
| 12 | `setCtxVar` | 技能上下文变量 ctxVar[key] = value | 无前端展示 | 否 |
| 13 | `pushPending` | 暂停引擎，等待用户输入 | 弹窗/选择界面 | 是（定义 pending 类型） |
| 14 | `popPending` | 恢复引擎执行 | 关闭弹窗 | 否 |
| 15 | `setPhase` | state.phase = phase | 阶段切换 UI | 否 |
| 16 | `nextPlayer` | 轮到下一位玩家 | 回合切换动画 | 否 |
| 17 | `judge` | 判定：翻牌→比较花色/点数 | 判定动画、判定牌展示 | 否 |
| 18 | `addPendingTrick` | 延时锦囊加入目标判定区 | 延时锦囊图标 | 否 |
| 19 | `removePendingTrick` | 延时锦囊从判定区移除 | 动画、结果展示 | 否 |
| 20 | `addTag` | player.tags 增加标记 | 标记展示 | 否 |
| 21 | `removeTag` | player.tags 移除标记 | 标记消失 | 否 |
| 22 | `kill` | player.info.alive = false | 死亡动画、武将牌翻面 | 否 |
| 23 | `gainCard` | card 从指定区域移到 player.hand | 获得牌动画 | 否 |
| 24 | `incrementKills` | 记录本回合使用【杀】次数 | 无前端展示 | 否 |
| 25 | `rearrangeDeck` | 重排牌堆顶/底 | 观星动画（展示牌→选择位置→确认） | 是（需要用户选择每张牌的位置） |

### 1.3 Atom 测试规范

每个 Atom 需要两种测试：

**后端测试：apply 状态转移**
```
describe('damage atom', () => {
  it('target health 减少 amount', () => {
    const state = createBaseState({ P1: { health: 3 } })
    const next = applyAtom(state, { type: 'damage', target: 'P1', amount: 2 })
    expect(next.players.P1.health).toBe(1)
  })
})
```

**前端测试：toEvents 视角分发**
```
describe('draw atom toEvents', () => {
  it('摸牌玩家看到牌面，其他人看到数量', () => {
    const [server, special, defaultEvt] = drawDef.toEvents(state, atom)
    expect(special.get('P1')?.cards).toHaveLength(2)  // P1 看到具体牌
    expect(defaultEvt?.cards).toBeUndefined()           // 他人只看数量
  })
})
```

## 二、Handler 纯化契约

### 2.1 契约定义

**每条规则都是强制的。违反任意一条即为架构违规。**

```
handler(ctx, state): SkillPhase[]
```

| # | 规则 | 禁止 | 允许 |
|---|---|---|---|
| R1 | Handler 必须是纯函数 | 调用 `emitEvent`、`applyAtom`、`applyDamage` | 仅返回 `SkillPhase[]` |
| R2 | Handler 可以读取 state，不可写入 state | 修改 state、调用 mutable 方法 | 通过 `ctx` 和 `state` 参数只读访问 |
| R3 | 状态变更必须通过 atoms | 直接修改 `state.players`、`state.zones` | 返回 `{ type: 'atoms', ops: [...] }` |
| R4 | 复杂流程必须用 phase 编排 | 在 handler 内写 for/while 遍历 | 使用 `condition`/`loop` phase |
| R5 | 用户交互必须用 prompt phase | 直接调用 UI 或设置 pending | 返回 `{ type: 'prompt', ... }` 由引擎调度 |

### 2.2 设计动机

Handler 纯化彻底解耦"决定要做什么"和"执行这些操作"：

```
技能触发 → handler(state, ctx) → SkillPhase[]  // 决策
              ↓
          executePlan(phases)  →  状态变更     // 执行
```

好处：
1. **可测试**：`handler(state, ctx)` 的输入和输出都是纯数据，无需 mock 引擎
2. **可追踪**：所有的 SkillPhase 构成了技能的"执行计划"，可以序列化、记录、回放
3. **可中断**：prompt phase 暂停后，resume 只需从上次位置继续执行剩余 phases
4. **可组合**：condition/loop 可以嵌套组合任意复杂逻辑

### 2.3 当前违规点及迁移方案

| 技能 | 违规 | 迁移方法 |
|---|---|---|
| 苦肉 | handler 使用 `checkDying` phase（实际上是合法的 phase） | 合法，无需迁移。但 engine 执行 checkDying 时需要发射 damageReceived 事件 |
| 观星 | handler 调用 `buildRearrangeTree()`（递归构建 phases） | 合法，纯函数递归构建 phases 树 |
| 青囊 | handler 在 phase 内使用 `setVar` 设置 usedThisTurn | 合法，通过 atoms 修改状态 |
| 反馈 | handler 返回两个 `atoms` phases（discardRandom → TODO gainCard） | 合法但 TODO 未完成 |

**当前真正的问题不是 handler 本身不纯，而是 engine 在 executePlan 时不发射 GameEvent。**

例如苦肉 handler 返回 `[{ type: 'atoms', ops: [{ type: 'damage', ... }] }]`，executePlan 跑完 damage atom 后应该自动触发 `damageReceived` 事件链，但当前代码不会——atoms 只是修改 state，不触发事件。

### 2.4 修复方案：Atom 执行后事件发射

```
executePlan(state, phases, ctx):
  for each phase:
    执行 phase（可能改变 state）
    if phase.type === 'atoms' && 某 atom 需要事件:
      emitEvent({ type: 'damageReceived', ... })  // 自动触发技能链
```

第 1 步定义映射：
```
const ATOM_TO_EVENTS = {
  damage:  [{ type: 'damageReceived', ... }],    // 触发 damage 相关的技能
  heal:    [{ type: 'healed', ... }],
  discard: [{ type: 'cardDiscarded', ... }],
  ...
}
```

第 2 步在 executePlan 中，处理完 atoms phase 后，根据 `ATOM_TO_EVENTS` 自动发射事件。

> ⚠️ 这需要与当前 handler（如 苦肉 手动用 `checkDying` phase）协调。
> 迁移方案见下文 §2.5。

### 2.5 迁移路线

```
阶段 1：文档对齐（本文档）
阶段 2：引擎改造 — executePlan 在 damage atom 后自动发射 damageReceived
阶段 3：检查 dying 机制 — damageReceived 触发濒死检查的时机
阶段 4：逐技能验证 — 每个 skill 的 handler 只返回 phases，不产生副作用
阶段 5：移除对 emitEvent 的 handler 内直接调用
```

## 三、场景测试 DSL

### 3.1 原则

- 不测 handler 返回值（那是 TS 类型检查的事）
- 不测单一步骤（那是 atom 单元测试的事）
- 测**完整的技能触发→决策→执行→状态变更**链路

场景 = `初始状态 → 操作 → 断言 → 操作 → 断言 → ...`

### 3.2 测试结构

```
tests/v2/scenarios/
  魏/
    奸雄.spec.ts        // 曹操受伤拿牌
    刚烈.spec.ts        // 夏侯惇判定+伤害来源选择
    遗计.spec.ts        // 郭嘉受伤摸2张+分配
  蜀/
    观星.spec.ts        // 诸葛亮准备阶段重排牌堆
    武圣.spec.ts        // 关羽红色牌当杀
  吴/
    苦肉.spec.ts        // 黄盖自伤摸牌+濒死
    制衡.spec.ts        // 孙权弃牌换牌
  群/
    闭月.spec.ts        // 貂蝉结束阶段摸牌
    青囊.spec.ts        // 华佗回合内限一次回血
  交互/
    多技能链.spec.ts    // 刘禅放权→张飞咆哮→杀
    濒死插入.spec.ts    // 苦肉濒死→华佗急救→继续摸牌
```

### 3.3 场景 DSL

```typescript
// 苦肉.spec.ts

import { scenario } from '../scenario-runner'

scenario('黄盖苦肉：自伤 1 摸 2')
  .setup(ctx => {
    ctx.selectCharacters('黄盖', '刘备')
    ctx.setHealth('P1', 4)
    ctx.enterPlayPhase()
  })
  .act('发动苦肉', (ctx) => {
    ctx.emitEvent('phaseBegin', { phase: '出牌', player: 'P1' })
  })
  .check('体力 -1', ctx => {
    expect(ctx.player('P1').health).toBe(3)
  })
  .check('摸 2 张牌', ctx => {
    expect(ctx.player('P1').hand.length).toBe(ctx.handSize('P1') - /* 原手牌 */ + 2)
  })
  .run()
```

```typescript
// 奸雄.spec.ts

scenario('曹操奸雄：受伤后获得造成伤害的牌')
  .setup(ctx => {
    ctx.selectCharacters('曹操', '刘备')
    ctx.registerTriggers('P1')
    ctx.giveCard('P2', '杀')
  })
  .act('P2 用杀攻击 P1', ctx => {
    const killId = ctx.findCard('P2', '杀')
    ctx.playCard('P2', killId, 'P1')
    ctx.respond('P1')  // P1 不出闪
  })
  .check('P1 受到 1 点伤害', ctx => {
    expect(ctx.player('P1').health).toBe(ctx.maxHealth('P1') - 1)
  })
  .check('奸雄触发：P1 获得杀', ctx => {
    expect(ctx.player('P1').hand).toContain(ctx.killCardId)
  })
  .run()
```

### 3.4 场景 Runner 接口

```typescript
interface ScenarioContext {
  state: GameState

  // Setup
  selectCharacters(...names: string[]): void
  setHealth(player: string, value: number): void
  setPhase(phase: TurnPhase): void
  enterPlayPhase(): void
  registerTriggers(player: string): void
  giveCard(to: string, cardName: string, count?: number): void

  // Actions
  playCard(player: string, cardId: string, target?: string): void
  respond(player: string, ...options: any[]): void
  useSkill(player: string, skillId: string): void
  discardCards(player: string, cardIds: string[]): void
  emitEvent(event: GameEvent): void

  // Queries
  player(name: string): PlayerState
  handSize(player: string): number
  findCard(player: string, cardName: string): string | undefined
  isPending(): boolean
  pendingType(): string | null

  // Verification helpers
  snapshot(): GameSnapshot       // 记录当前状态快照
  diff(snapshot): StateDiff      // 对比状态变化
}
```

## 四、前端 Atom 测试

### 4.1 原则

- 每个 atom 类型对应独立的前端测试
- 测试"给定 atom → 前端正确响应"
- 前端响应包括：状态变更、UI 交互、媒体播放

### 4.2 测试分类

| 测试类型 | 内容 | 工具 |
|---|---|---|
| **Atom 到状态** | `atom.toEvents()` → 每个玩家收到正确的事件 | vitest |
| **事件到 UI** | 事件 → React 组件渲染正确的 UI | vitest + testing-library |
| **交互到 pending** | pending → 弹窗/选牌/确认 → 返回 action | vitest + testing-library |
| **动画/音效** | atom 触发了正确的媒体播放 | vitest (mock) |

### 4.3 示例

```typescript
// frontend/tests/atoms/draw.test.tsx

describe('draw atom 前端效果', () => {
  it('摸牌玩家看到手牌新增 + 牌面', () => {
    const events = drawDef.toEvents(state, { type: 'draw', player: 'P1', count: 2 })
    const p1Event = specialMap.get('P1')
    expect(p1Event?.cards).toHaveLength(2)       // 看到牌面
    expect(p1Event?.type).toBe('draw')
  })

  it('其他玩家只看数量变化', () => {
    const events = drawDef.toEvents(state, { type: 'draw', player: 'P1', count: 2 })
    const defaultEvent = events[2]
    expect(defaultEvent?.cards).toBeUndefined()   // 看不到牌
    expect(defaultEvent?.count).toBe(2)           // 只看数量
  })
})
```

### 4.4 前端 Atom 注册

每个新 atom 必须注册以下信息（供前端自动处理）：

```typescript
interface FrontendAtomRegistration {
  type: string
  /** 此 atom 是否需要用户交互（如选牌） */
  interactive: boolean
  /** 交互类型（如果 interactive） */
  interactionType?: 'selectCards' | 'selectPlayer' | 'confirm' | ...
  /** 动画组件 */
  animationComponent?: React.ComponentType<AnimationProps>
  /** 音效 ID */
  sound?: string
}
```

> 注：前端 atom 注册是后续步骤，当前重点工作是先完善后端 atom 清单和 handler 规范。
